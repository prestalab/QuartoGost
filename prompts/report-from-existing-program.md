# Промпт для агента: отчёт о НИР по готовой программе

Замените значения в угловых скобках перед использованием:

- `<PROGRAM_REPO_PATH>` — путь к репозиторию или рабочей папке программы;
- `<PROJECT_SLUG>` — короткое имя проекта латиницей;
- `<REPORT_NAME>` — базовое имя выходного файла.

```text
Используй проект QuartoGost в `C:\projects\QuartoGost` для подготовки отчёта о НИР по уже реализованной программе из каталога `<PROGRAM_REPO_PATH>`.

Контекст:
- программа уже написана;
- ты знаешь структуру этой программы, её модули, сценарии запуска и назначение;
- нужно подготовить именно отчёт по ГОСТ 7.32-2017;
- результат должен собираться средствами QuartoGost под Windows.

Обязательно используй:
- шаблон:
  `C:\projects\QuartoGost\templates\report\report-template.qmd`
- пример заполнения:
  `C:\projects\QuartoGost\examples\report-demo\main.qmd`
- skills проекта:
  `quarto-gost-workflow`
  `quarto-gost-scientific-editor`
  `quarto-gost-report`

Что нужно сделать:
1. Изучи код, структуру каталогов, конфигурацию, зависимости, точки входа, входные и выходные данные программы в `<PROGRAM_REPO_PATH>`.
2. Подготовь исходный `qmd`-документ отчёта на основе шаблона QuartoGost.
3. Используй пример `report-demo` как образец:
   - структуры;
   - заполнения блока `gost:`;
   - кросс-ссылок;
   - таблиц, рисунков, формул;
   - автоматической библиографии через `::: {#refs}`.
4. Не выдумывай функциональность, архитектуру, алгоритмы, результаты тестов, метрики, нормативные основания, публикации и источники.
5. Если каких-то данных нет в коде или документации, оставляй явные пометки `TODO` или `Требует уточнения`.
6. Итоговый файл сохрани как:
   `C:\projects\QuartoGost\work\report\<PROJECT_SLUG>\<REPORT_NAME>.qmd`
7. Если для отчёта нужны дополнительные файлы `.bib`, изображения или диаграммы, создай их рядом с документом в той же рабочей папке.

Требования к содержанию:
- текст должен опираться на реальные данные из кода и репозитория программы;
- стиль изложения должен быть деловым, техническим и соответствовать отчёту о НИР;
- структура должна соответствовать ГОСТ 7.32-2017;
- должны использоваться кросс-ссылки на рисунки, таблицы и формулы;
- список литературы не оформляй вручную: используй `.bib`, ссылки `[@key]` и вывод через `::: {#refs}`.

Рекомендуемая структура отчёта:
- реферат;
- перечень сокращений и обозначений;
- введение;
- анализ предметной области и постановка задачи;
- назначение и область применения программы;
- архитектура и состав программного средства;
- используемые алгоритмы, модели, данные и зависимости;
- реализация и особенности эксплуатации;
- результаты тестирования, апробации или применения;
- заключение;
- список использованных источников;
- приложения.

Работа с титульными данными:
- заполни блок `gost:` только теми реквизитами, которые действительно меняются;
- не трогай постоянный текст формы;
- если точные реквизиты организации неизвестны, оставь аккуратные плейсхолдеры для ручного заполнения.

Если уместно, добавь:
- таблицу состава модулей;
- таблицу внешних зависимостей;
- рисунок архитектуры;
- таблицу сценариев запуска;
- графики или диаграммы, если они действительно выводятся из данных проекта или результатов работы программы.

В конце работы:
- перечисли, какие файлы созданы или изменены;
- перечисли, что нужно дозаполнить вручную;
- если запускал сборку, приведи точную команду сборки.
```
