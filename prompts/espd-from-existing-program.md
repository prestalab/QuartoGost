# Промпт для агента: документация ЕСПД по готовой программе

Замените значения в угловых скобках перед использованием:

- `<PROGRAM_REPO_PATH>` — путь к репозиторию или рабочей папке программы;
- `<PROJECT_SLUG>` — короткое имя проекта латиницей;
- `<DOC_NAME>` — базовое имя выходного файла;
- `<DOC_KIND>` — вид документа, например `Описание программы`, `Руководство программиста`, `Текст программы`.

```text
Используй проект QuartoGost в `C:\projects\QuartoGost` для подготовки документации ЕСПД по уже реализованной программе из каталога `<PROGRAM_REPO_PATH>`.

Контекст:
- программа уже написана;
- ты знаешь её структуру, состав модулей, назначение и способ запуска;
- нужно подготовить документ в логике ГОСТ 19.xxx / ГОСТ 2.105-95;
- документ должен собираться средствами QuartoGost под Windows.

Обязательно используй:
- шаблон:
  `C:\projects\QuartoGost\templates\espd\espd-template.qmd`
- пример заполнения:
  `C:\projects\QuartoGost\examples\espd-demo\main.qmd`
- skills проекта:
  `quarto-gost-workflow`
  `quarto-gost-scientific-editor`
  `quarto-gost-espd`

Что нужно сделать:
1. Изучи код, структуру каталогов, конфигурацию, зависимости, сценарии запуска, входные и выходные данные программы в `<PROGRAM_REPO_PATH>`.
2. Определи, какой вид документа ЕСПД лучше подходит для этой задачи.
3. Если выбор неоднозначен, по умолчанию делай `<DOC_KIND>` как `Описание программы`.
4. Подготовь исходный `qmd`-документ на основе шаблона QuartoGost.
5. Используй пример `espd-demo` как образец:
   - заполнения блока `gost:`;
   - оформления разделов и подразделов;
   - кросс-ссылок;
   - таблиц, рисунков, формул;
   - автоматической библиографии.
6. Не выдумывай функциональность, интерфейсы, режимы, ограничения, данные, алгоритмы и результаты испытаний.
7. Если каких-то данных не хватает, оставляй явные пометки `TODO` или `Требует уточнения`.
8. Итоговый файл сохрани как:
   `C:\projects\QuartoGost\work\espd\<PROJECT_SLUG>\<DOC_NAME>.qmd`
9. Если нужны дополнительные `.bib`, изображения, схемы или вспомогательные файлы, создай их рядом с документом в той же рабочей папке.

Требования к содержанию:
- текст должен опираться только на реально реализованную программу;
- структура должна соответствовать логике ЕСПД и ГОСТ 2.105-95;
- разделы, подразделы, пункты и подпункты должны быть упорядочены и технически последовательны;
- список литературы не оформляй вручную: используй `.bib`, ссылки `[@key]` и вывод через `::: {#refs}`;
- если используются рисунки, таблицы и формулы, должны быть корректные ссылки на них в тексте.

Рекомендуемая структура для `Описание программы`:
- назначение программы;
- условия применения;
- описание задачи;
- входные данные;
- выходные данные;
- описание алгоритма;
- используемые технические и программные средства;
- сообщения пользователю;
- требования к эксплуатации;
- приложения.

Работа с титульными данными:
- заполни блок `gost:` только переменными реквизитами;
- не заменяй постоянный текст формы на плейсхолдеры;
- если реквизиты организации неизвестны, оставь аккуратные значения для последующего ручного заполнения.

Если уместно, добавь:
- таблицу модулей и их назначения;
- таблицу входных и выходных файлов;
- схему архитектуры;
- описание сценария запуска;
- список ограничений и условий корректной работы.

В конце работы:
- перечисли, какие файлы созданы или изменены;
- перечисли, какие поля нужно дозаполнить вручную;
- если запускал сборку, приведи точную команду сборки.
```
